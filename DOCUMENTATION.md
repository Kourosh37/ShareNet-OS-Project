# مستندات ShareNet-Phase1-C

## 1. معرفی پروژه

ShareNet یک سیستم ساده اشتراک فایل در شبکه داخلی است. این نسخه فاز اول پروژه است و هدف آن نمایش مفاهیم پایه سیستم‌عامل و شبکه مانند سوکت TCP، انتقال فایل، تقسیم فایل به chunk و مدیریت خطا است.

## 2. هدف فاز اول

هدف فاز اول ساخت یک نسخه ساده، قابل فهم و قابل اجرا است. سرور فایل‌ها را نگهداری می‌کند و کلاینت می‌تواند لیست فایل‌ها را بگیرد، فایل آپلود کند و فایل دانلود کند.

## 3. معماری کلی سیستم

سیستم از یک سرور مرکزی و چند کلاینت تشکیل شده است. در این فاز سرور هم‌زمان چند درخواست را پردازش نمی‌کند. هر کلاینت برای هر عملیات یک اتصال TCP جداگانه باز می‌کند، درخواست خود را انجام می‌دهد و اتصال را می‌بندد.

نسخه گرافیکی با raylib روی همین منطق ساخته شده است. GUI سرور یک پنجره برای شروع و توقف سرور، مشاهده فایل‌ها و دیدن log دارد. GUI کلاینت فرم‌های ساده برای آپلود، دانلود و گرفتن لیست فایل‌ها دارد. هسته انتقال فایل و پروتکل در هر دو نسخه مشترک است.

## 4. نقش Server

سرور روی پورت تعریف شده در `config.h` گوش می‌دهد. پس از اتصال یک کلاینت، فقط همان درخواست را پردازش می‌کند. فایل‌های آپلود شده در پوشه `server_files` ذخیره می‌شوند و دانلودها از همین پوشه انجام می‌شوند.

## 5. نقش Client

کلاینت یک منوی ساده متنی دارد. کاربر می‌تواند لیست فایل‌ها را ببیند، مسیر فایل را برای آپلود وارد کند یا نام فایل را برای دانلود وارد کند. فایل‌های دانلود شده در پوشه `downloads` ذخیره می‌شوند.

## 6. دلیل single-threaded بودن سرور در فاز اول

در فاز اول هدف تمرکز روی پروتکل، انتقال درست داده و ساختار ماژولار است. حذف thread و fork باعث می‌شود رفتار برنامه ساده‌تر، قابل ردیابی‌تر و مناسب‌تر برای آموزش باشد.

در نسخه گرافیکی سرور نیز از thread یا fork استفاده نشده است. سوکت accept به صورت non-blocking تنظیم می‌شود تا پنجره raylib زنده بماند. وقتی یک کلاینت وصل شود، درخواست همان کلاینت به صورت ترتیبی پردازش می‌شود و سپس سرور دوباره آماده کلاینت بعدی می‌شود.

## 7. ساختار پوشه‌ها و فایل‌ها

- `include`: هدرهای عمومی پروژه
- `src/common`: کدهای مشترک بین سرور و کلاینت
- `src/server`: کدهای مخصوص سرور
- `src/client`: کدهای مخصوص کلاینت
- `server_files`: فایل‌های ذخیره شده روی سرور
- `client_files`: فایل‌های آماده برای آپلود
- `downloads`: فایل‌های دانلود شده
- `test_files`: فایل‌های نمونه تست

## 8. توضیح کامل هر فایل

- `include/config.h`: تنظیمات اصلی مانند IP، پورت، اندازه chunk و مسیر پوشه‌ها
- `include/protocol.h`: نوع پیام‌ها، ساختار `MessageHeader` و توابع پروتکل
- `include/socket_utils.h`: توابع ساخت و مدیریت سوکت
- `include/file_utils.h`: توابع کار با فایل و مسیر
- `include/chunk.h`: ساختار `ChunkHeader` و توابع انتقال chunk
- `include/server.h`: واسط توابع سرور
- `include/client.h`: واسط توابع کلاینت
- `src/common/protocol.c`: ارسال و دریافت header و payload متنی
- `src/common/socket_utils.c`: پیاده‌سازی TCP socket و ارسال/دریافت کامل
- `src/common/file_utils.c`: ساخت پوشه، بررسی نام فایل، لیست فایل‌ها و اندازه فایل
- `src/common/chunk.c`: تقسیم فایل به chunk و بازسازی فایل
- `src/server/server_main.c`: نقطه شروع سرور
- `src/server/server.c`: منطق درخواست‌های سرور
- `src/client/client_main.c`: نقطه شروع کلاینت
- `src/client/client.c`: منوی کلاینت و عملیات list/upload/download
- `src/gui/server_gui.c`: رابط گرافیکی raylib برای سرور
- `src/gui/client_gui.c`: رابط گرافیکی raylib برای کلاینت
- `scripts/build-linux.sh`: ساخت خروجی‌های Linux
- `scripts/build-windows.ps1`: ساخت خروجی‌های Windows
- `scripts/build-windows.bat`: اجرای ساده اسکریپت PowerShell در Windows
- `scripts/build-macos.sh`: ساخت خروجی‌های macOS

## 9. پروتکل ارتباطی

پروتکل binary است و JSON یا متن ساختاریافته استفاده نمی‌کند. هر پیام با یک `MessageHeader` شروع می‌شود. اگر پیام payload داشته باشد، بلافاصله بعد از header ارسال می‌شود.

## 10. ساختار MessageHeader

```c
typedef struct {
    uint32_t type;
    uint32_t payload_size;
} MessageHeader;
```

هر دو فیلد قبل از ارسال با `htonl` به network byte order تبدیل می‌شوند و پس از دریافت با `ntohl` به حالت میزبان برمی‌گردند.

## 11. ساختار ChunkHeader

```c
typedef struct {
    uint32_t chunk_index;
    uint32_t total_chunks;
    uint32_t chunk_size;
} ChunkHeader;
```

این header قبل از داده واقعی هر chunk ارسال می‌شود. همه فیلدهای آن نیز با `htonl` و `ntohl` تبدیل می‌شوند.

## 12. روند دریافت لیست فایل‌ها

کلاینت یک اتصال TCP باز می‌کند و پیام `MSG_REQUEST_LIST` را می‌فرستد. سرور فایل‌های موجود در `server_files` را می‌خواند و با `MSG_LIST_RESPONSE` یک متن ساده شامل نام، اندازه و زمان تغییر فایل‌ها برمی‌گرداند.

## 13. روند آپلود فایل

کلاینت ابتدا metadata را با پیام `MSG_REQUEST_UPLOAD` می‌فرستد. payload شامل طول نام فایل، اندازه فایل ۶۴ بیتی و بایت‌های نام فایل است. سپس فایل با پیام‌های `MSG_CHUNK_DATA` ارسال می‌شود. در پایان `MSG_DONE_TRANSFER` فرستاده می‌شود و سرور پیام `MSG_UPLOAD_RESPONSE` برمی‌گرداند.

## 14. روند دانلود فایل

کلاینت نام فایل را با پیام `MSG_DOWNLOAD_REQUEST` ارسال می‌کند. سرور نام را بررسی می‌کند. اگر فایل وجود نداشته باشد `MSG_ERROR` برمی‌گرداند. اگر فایل موجود باشد، داده‌ها را chunk به chunk با `MSG_CHUNK_DATA` و سپس `MSG_DONE_TRANSFER` ارسال می‌کند.

## 15. روند تقسیم فایل به chunk

تابع `calculate_total_chunks` تعداد chunkها را از اندازه فایل و `CHUNK_SIZE` حساب می‌کند. تابع `send_file_chunks` هر بار بخشی از فایل را می‌خواند و به همراه `ChunkHeader` می‌فرستد.

## 16. روند بازسازی فایل نهایی

گیرنده هر `ChunkHeader` را می‌خواند، شماره chunk را بررسی می‌کند و داده را به ترتیب در فایل خروجی می‌نویسد. اگر شماره chunk با مقدار مورد انتظار برابر نباشد، انتقال نامعتبر در نظر گرفته می‌شود.

## 17. مدیریت خطاها

خطاهای مهم با `MSG_ERROR` به طرف مقابل اعلام می‌شوند. خطاهای سیستمی مانند شکست socket، bind، connect، fopen یا stat در خروجی استاندارد خطا گزارش می‌شوند.

## 18. نکات امنیتی ساده

تابع `is_safe_filename` جلوی نام فایل خالی، مسیر شامل `/` یا `\` و عبارت `..` را می‌گیرد. این کار از path traversal ساده جلوگیری می‌کند و اجازه نمی‌دهد کلاینت بیرون از پوشه‌های مجاز فایل بنویسد یا بخواند.

## 19. نحوه کامپایل

```sh
make
```

برای پاک کردن خروجی‌ها:

```sh
make clean
```

## 20. نحوه اجرا

در ترمینال اول:

```sh
./sharenet_server
```

در ترمینال دوم:

```sh
./sharenet_client
```

برای نسخه گرافیکی:

```sh
make gui
./sharenet_server_gui
./sharenet_client_gui
```

در پنجره سرور روی `Start Server` کلیک کنید. سپس در پنجره کلاینت عملیات مورد نظر را با دکمه‌ها انجام دهید.

برای ساخت خروجی جداگانه هر سیستم‌عامل:

```sh
./scripts/build-linux.sh
./scripts/build-macos.sh
```

روی Windows:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\build-windows.ps1
```

خروجی‌ها در پوشه `dist` ساخته می‌شوند. برای ساخت GUI باید raylib نصب و قابل لینک باشد.

## 21. سناریوی دمو

ابتدا سرور را اجرا کنید. سپس کلاینت را اجرا کنید. گزینه آپلود را انتخاب کنید و `client_files/sample.txt` را وارد کنید. بعد گزینه لیست فایل‌ها را اجرا کنید تا `sample.txt` را ببینید. سپس گزینه دانلود را اجرا کنید و نام `sample.txt` را وارد کنید.

## 22. روش بررسی صحت فایل

```sh
diff client_files/sample.txt downloads/sample.txt
sha256sum client_files/sample.txt downloads/sample.txt
```

نبود خروجی برای `diff` یعنی دو فایل یکسان هستند. برابر بودن hashها نیز همین موضوع را تایید می‌کند.

## 23. محدودیت‌های فاز اول

- سرور هم‌زمان فقط یک کلاینت را پردازش می‌کند.
- احراز هویت وجود ندارد.
- رمزنگاری وجود ندارد.
- امکان resume برای دانلود یا آپلود ناقص وجود ندارد.
- پروتکل برای سادگی فقط نیازهای فاز اول را پوشش می‌دهد.
- عملیات انتقال فایل در GUI هنگام پردازش یک درخواست می‌تواند برای مدت کوتاه پنجره را مشغول کند، چون عمدا thread جداگانه استفاده نشده است.

## 24. پیشنهاد برای توسعه در فازهای بعدی

- پشتیبانی از چند کلاینت با thread یا event loop
- افزودن احراز هویت کاربر
- استفاده از TLS برای رمزنگاری ارتباط
- افزودن checksum برای هر فایل
- پشتیبانی از resume
- اضافه کردن لاگ ساخت‌یافته
- طراحی پروتکل نسخه‌دار برای سازگاری آینده
